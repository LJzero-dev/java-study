상속(inheritance)
 - 다른 클래스로 부터 상속받아 그 클래스의 기능(메소드)과 속성(변수, 상수)을 사용하는 것
 	단, 소스의 재활용적인 측면으로만 상속을 사용하면 안됨

 - 자바에서는 단일 상속만을 지원하기 떄문에 하나의 클래스 에서만 상속을 받을 수 있음(c언어는 다중상속)
 	자바에서는 기본적으로 모든 클래스(Object 클래스 제외)가 이미 상속받고 있음

 - 자바에서의 상속은 상위클래스와 동일한 성격을 지닌 클래스로 생성하기 위해 상속을 사용함
 	상속을 받은 클래스를 '하위클래스' 또는 '서브(sub)클래스', 상속을 해준 클래스를 '상위클래스' 또는 '슈퍼(super)클래스' 라고 함

 - private으로 선언된 멤버도 상속은 되지만 직접적인 접근은 불가(메소드를 통해 접근해야 함)

 - 상위클래스에서 하위클래스로의 접근은 불가

 - 문법
class 하위클래스명 extends 상위클래스명 {
	하위클래스생성자() {
		super();	//	상위클래스의 생성자를 호출하는 것으로 반드시 하위클래스 생성자의 첫줄에 있어야 함
	}
}

상속을 하기 위한 클래스간의 관계
 - 기본적으로 모든 클래스(final 클래스는 제외)를 상속받을 수 있으나, 특정 관계가 성립하는 클래스들 끼리 상속을 해야 자바에서의 상속 목적에 부합함
 - is-a 관계
 	'하위클래스 is a 상위클래스'의 관계가 성립되는 클래스들을 의미
	즉, 하위클래스가 상위클래스의 일종으로 사용 될 경우를 의미

 - has-a 관계
 	'하위클래스 has a 상위클래스'의 관계가 성립되는 클래스들을 의미
	하위클래스가 상위클래스를 소유하고 있는 경우를 의미
	단, 소유하지 않는 경우가 있을 경우 상속은 다시 생각해봐야 함(예외적인 상황이 있을 경우 상속을 받지 않는 것이 좋음)
	만약 소유하지 않는 경우가 있는데도 상속을 하면 소유하지 않은 인스턴스는 표현이 안됨
	소유하지 않는 경우가 있는 has-a 관계는 상속이 아닌 복합방식으로 처리해야 함

* null : 빈 값을 의미하는 것으로 객체가 비었을 경우 들어가는 아무것도 아닌 값
 - 객체가 비어있는 것을 의미하는 값으로 if문 등에서 조건으로 사용할 수 있음
 - 조건으로 사용 시 비교연산자인 '=='로 비교할 수 있음
 - 예) 객체 == null -> 객체가 비었는지 여부를 검사하는 조건식

오버라이딩(overriding)
 - 상속관계의 클래스 중 하위클래스에서 상위클래스의 특정 메소드를 재정의 하는 것
 	오버로딩과 다르게 선언부(리턴타입, 메소드명, 매개변수)가 완전히 동일해야 함
 - 오버라이딩 시 접근제어지시자는 반드시 원본보다 넓거나 최소 같은 범위를 사용해야 함
 	원본이 public인 경우 오버라이딩되는 메소드는 무조건 public으로 선언해야 함

- 하위클래스의 인스턴스로 오버라이딩된 메소드를 호출하면 무조건 하위 클래스의 메소드가 실행됨
	하위클래스의 인스턴스로는 상위클래스의 원본 메소드를 호출할 수 없음
	상위클래스의 오버라이딩된 원본 메소드를 호출하려면 오버라이딩된 메소드에서 super 명령어로 호출해야 함
	예) super.원본메소드명();

※ 상위클래스로 선언된 인스턴스에는 하위클래스의 인스턴스가 저장될 수 있음
 - 그 인스턴스는 알맹이는 하위클래스지만 JVM에서는 상위클래스로 인식됨
 	그러므로 실제 해당 인스턴스로 사용할 수 있는 멤버는 상위클래스에 있는 멤버들만 가능
 - 이 상황에서 하위클래스의 멤버들을 사용하는 방법
 	1. 오버라이딩 - 필요한 메소드를 상위클레스에도 정의하여 하위클래스에서 오버라이딩하게 하여 사용
	 - 단, 상위클래스를 변경할 수 있으며, 변경해도 괜찮은 경우에만 사용가능
	2. 형변환 - 해당 인스턴스를 하위클래스로 형변환한 후 사용
	 - Java API에서 제공하는 클래스이거나 변경하면 안되는 상위클래스일 경우
	 
Object 클래스
 - java.leng.Object 클래스는 모든 자바의 클래스들이 상속받고 있는 최상위 클래스
 	extends Object 명령은 모든 클래스에서 생략되어 있을 뿐 Object 클래스를 상속받고 있음
 - 모든 클래스는 Object클래스의 하위클래스로 Object형으로 선언한 인스턴스에는 모든 종류의 인스턴스가 들어갈 수 있음
 	인스턴스가 아닌 기본 자료형 들도 Wrapper 클래스들을 이용하면 인스턴스화 하여 들어갈 수 있음
 - 모든 클래스에서는 Object클래스의 멤버들을 자유롭게 사용할 수 있음(예 : equals(), toString() 등)
 	toString() 메소드 : Java API에서 오버라이딩하여 사용하라고 권장하는 메소드
	 - 객체(인스턴스)를 출력할 때 그 객체를 문자열로 변환시켜주는 메소드
	 - 객체 출력 시 자동으로 실행되는 메소드로 객체를 문자열로 변환시키는 데 그 값이 쓸모가 없으므로 원하는 형태의 문자열로 오버라이딩하여 사용하는것이 좋음
	 - 주로 정보저장용 클래스에서 많이 사용되며, 프로그래밍시 확인(디버깅)용으로 사용됨

final 명령
 - final 변수 : 해당 변수를 상수로 생성 - 값 초기화 후 변경 불가
 	선언 : 접근제어지시자 final 자료형 변수명;
 - final 메소드 : 오버라이딩 불가(재정의 하지 않고 있는 그대로 메소드를 사용하라는 의미)
 	선언 : 접근지시제어자 final 리턴타입 메소드명() {}
 - final 클래스 : 상속 불가 - 상속받지 말고 인스턴스를 생성하여 사용하라는 의미(주로 java에서 제공하는 클래스 들)
 	선언 : 접근제어지시자 final class 클래스명 {}
	
abstract 클래스
 - 추상(화) 클래스라고도 하는 클래스의 일종으로 미완성 클래스
 - 하나 이상의 abstract 메소드를 가지고 있는 클래스로 인스턴스 생성이 불가능
 	abstract 메소드 : 미완성된 메소드로 구현부(중괄호) 없이 선언부만 존재하는 메소드
		- 선언 : 접근제어지시자 abstract 리턴타입 메소드명([매개변수]);
 - 상속에서 상위클래스의 역할을 하기 위해 정의되는 클래스 
 	문법 : abstract class 클래스명 { ... };
 - abstract 클래스 사용 시 지켜야할 규칙
 	abstract 클래스를 상속받은 하위클래스에서는 반드시 abstract 메소드를 오버라이딩하여 완성해야 함
	만약 abstract 메소드를 오버라이딩하지 않거나 완성하지 않으면 하위클래스도 abstract로 선언해야 함
 - 단점 : abstract 클래스를 사용하려면 반드시 상속받아야 하는 단점이 있어 사용빈도가 높지 않음

interface
 - 모든 메소드가 abstract 메소드인 완전한 abstract 클래스 interface로 지정할 수 있음
 	인터페이스는 클래스는 아니나 클래스처럼 동작하기도 함
	문법 : interface 인터페이스 이름 { ... }

 - 자바에서는 불가능한 다중 상속의 효과를 낼 수 있는 방법으로 사용되기도 함
 	상속의 extends가 아닌 구현의 implements 명령을 사용하며, 여러 개의 인터페이스를 하나의 클래스에서 implements 할 수도 있음
	인터페이스 끼리는 상속(extends)도 가능함

 - 인터페이스에서 선언하는 모든 메소드는 자동으로 public abstract로 선언됨
 	인터페이스의 메소드를 오버라이딩 할 경우 무조건 public으로 선언해야 함

 - 인터페이스에서 선언하는 모든 변수는 자동으로 public static final로 선언됨
 	공용으로 사용할 상수때문에 interface를 사용하기도 함
